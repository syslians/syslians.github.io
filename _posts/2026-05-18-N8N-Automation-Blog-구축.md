# 블로그 자동화 아키텍처

## 1. 프로젝트의 목적

저는 현재 GitHub 기반 기술 블로그를 운영하고 있습니다. 기존에는 VS Code 같은 문서 편집기에서 Markdown으로 글을 작성한 뒤, GitHub 레포지터리의 _posts 하위 디렉터리에 yyyy-mm-dd-title.md 형식으로 파일을 추가했습니다. 이후 commit을 하면 CI/CD workflow가 실행되고, GitHub Pages에 배포되어 블로그에 반영되는 구조였습니다.

이 방식은 단순하지만, 글을 쓸 때마다 파일을 만들고, 이미지를 정리하고, GitHub 레포지터리에 들어가 commit까지 해야 해서 반복 작업이 꽤 번거로웠습니다. 그래서 Notion에서 글을 작성하면 GitHub 블로그 포스트로 자동 발행되는 워크플로우를 n8n으로 구축하게 되었습니다.

이번 자동화의 목표는 명확했습니다.

Notion을 글쓰기 도구로 사용하고, n8n이 Notion 데이터를 Markdown으로 변환한 뒤, 최종 결과물을 GitHub Pages 블로그에 올리는 것입니다.

n8n을 사용하면서 가장 크게 느낀 장점은 아키텍처 구성이 직관적이라는 점이었습니다. 일반적으로 여러 API를 코드로 연결하려면 요청, 응답, 인증, 데이터 변환 흐름을 직접 설계해야 합니다. n8n에서는 이 흐름을 노드 단위로 나누고 연결할 수 있어서, 전체 구조를 눈으로 확인하면서 작업할 수 있었습니다.

물론 모든 프로젝트에 n8n이 적합하다고 보지는 않습니다. 다만 데이터를 가져오고, 가공하고, 다른 API로 전달하는 ETL 유형의 작업이나, 여러 API를 연결해야 하거나, 파이프라인이 비교적 선형적이고 길이가 너무 길지 않은 작업과 같은 자동화, 멀티-에이전트 오케스트레이션 작업에는 앞으로도 자주 사용할 수 있을 것 같습니다.

비슷한 유형의 오픈소스 툴로는 Lang graph, Flowise 가 있다고 하는데 저는 아직 사용해본적은 없습니다. 추후에 사용하게 되면 포스팅 하겠습니다.

## 2. 각 노드들에 대한 간단한 설명

Notion Trigger는 Notion 데이터베이스의 페이지가 수정되었는지 주기적으로 확인하는 시작 노드입니다. 블로그 글이 작성되거나 업데이트되면 이 노드를 통해 워크플로우가 시작됩니다. 현재는 N8N 페이지에서 execute workflow 버튼을 눌러서 실행하고 있습니다. cron job으로 실행하게 하거나, 이벤트에 반응하게 만들수 있지만 현재 수준에서는 리소스 낭비라고 판단하여 수동으로 워크플로우를 실행시킬수 있도록 만들었습니다.

Search a page는 수정된 Notion 페이지의 정보를 검색합니다. 제목, 카테고리, 작성자, 발행 여부 같은 메타데이터를 가져오는 역할을 합니다.

Get many child blocks는 Notion 페이지 안의 본문 블록을 가져옵니다. Notion은 문서를 블록 단위로 다루기 때문에, 본문을 Markdown으로 바꾸려면 각 블록을 먼저 수집해야 합니다. 이 노드에서는 중첩 블록도 함께 가져오도록 설정했습니다.

Metadata Normalization은 Notion에서 가져온 메타데이터를 블로그에서 쓰기 좋은 형태로 정리합니다. 제목, slug, 날짜, 카테고리, 작성자, 발행 여부 등을 변환합니다.

block chunking은 가져온 Notion 블록들을 하나의 blocks 배열로 묶습니다.

Merge block and Metadata는 메타데이터와 본문 블록 데이터를 하나로 합칩니다. 이후 단계에서는 이 병합된 데이터를 기준으로 Markdown 변환을 진행합니다.

Notion expression Normalization to Markdown expression은 이 워크플로우의 핵심 노드입니다. Notion 블록을 Markdown 문법으로 변환합니다. 문단, 제목, 리스트, 코드 블록, 이미지, embed, table 등을 처리하고, Jekyll에서 사용할 frontmatter도 함께 생성합니다.

Split Out, urlNormalize and create UUID, Split Out1은 본문에서 추출된 이미지 URL을 분리하고, GitHub에 저장할 이미지 파일명을 만드는 단계입니다.

If (image is is Temporary)는 이미지 URL이 Notion의 임시 presigned URL인지 확인합니다. Notion에서 제공하는 이미지 URL은 일정 시간이 지나면 만료될 수 있기 때문에, 이런 경우에는 이미지를 따로 저장해야 합니다.

HTTP Request2는 임시 이미지 URL에서 실제 이미지 파일을 다운로드합니다.

Create a image FIle은 다운로드한 이미지를 GitHub 저장소의 assets 경로에 업로드합니다.

Split Out2와 Merge1은 이미지 처리 결과와 원래 Markdown 데이터를 다시 합칩니다.

imageIngection은 Markdown 안에 임시로 넣어둔 !/assets/image_c68bc373-c908-4f1e-b631-943d8514b040.png placeholder를 실제 이미지 경로로 바꿉니다.

content encoding(base64)는 최종 Markdown 내용을 GitHub API에 업로드할 수 있도록 base64로 인코딩합니다.

HTTP Request1은 GitHub API를 호출해 _posts 디렉터리에 Markdown 파일을 생성합니다.

Discord는 게시글 생성이 끝난 뒤 Discord로 알림을 보내는 노드입니다.

## 3. 전체 흐름

전체 흐름은 크게 다섯 단계로 볼 수 있습니다.

### 첫 번째는 Notion 변경 감지입니다.

Notion 데이터베이스에 블로그 글을 작성하고 n8n 트리거가 동작하면, 워크플로우가 시작됩니다.

### 두 번째는 콘텐츠 수집입니다.

Notion 페이지는 메타데이터와 본문으로 나뉩니다. 메타데이터에는 제목, 카테고리, 작성자, 발행 여부 같은 정보가 들어 있고, 본문은 Notion 블록으로 구성됩니다. 본문 블록 안에는 텍스트뿐 아니라 이미지, LaTeX, code, embed 같은 요소도 포함될 수 있습니다.

### 세 번째는 변환과 정규화입니다.

가져온 메타데이터는 GitHub Pages 블로그 포맷에 맞게 정리합니다. 본문 역시 Markdown으로 변환합니다. 이 과정에서 Notion API 응답이 Markdown과 바로 호환되지 않는 부분들이 있어, function node에서 JavaScript로 직접 변환 로직을 작성했습니다. 결국 HTML이나 Markdown 문자열을 다루는 작업이 많았기 때문에 JavaScript가 가장 편했습니다.

이미지는 별도 처리가 필요했습니다. Notion의 이미지 URL이 presigned URL인 경우 그대로 사용하면 시간이 지나 접근할 수 없게 됩니다. 그래서 이런 이미지는 다운로드한 뒤 제 GitHub 레포지터리의 assets 경로에 저장하고, Markdown에서는 해당 경로를 참조하도록 바꿨습니다. 반대로 일반 URL처럼 계속 접근할 수 있는 이미지는 원본 URL을 그대로 사용하도록 처리했습니다.

### 네 번째는 GitHub 발행입니다.

최종 Markdown 파일을 base64로 직렬화한 뒤 GitHub API를 통해 블로그 레포지터리의 _posts 하위에 생성합니다. 이후에는 기존과 동일하게 GitHub의 CI/CD workflow가 실행되고, GitHub Pages에 배포되면서 블로그에 반영됩니다.

정리하면, Notion에서 글을 작성하고 발행 상태를 켜면 n8n이 글을 가져와 Markdown으로 변환하고, 이미지까지 정리한 뒤 GitHub Pages 블로그에 자동으로 포스팅하는 구조입니다.

### 다섯번째는 최종 포스팅 결과 discord channel에 전송입니다.

디스코드 노드는 단순히 ci-cd 워크플로우의 결과에 따라 포스팅이 성공했는지, 실패했는지의 결과를 전송하기만 하여 결과를 확인할 수 있게만 만들어두었습니다.

## 4. 회고 및 고도화할 부분

이번 작업은 n8n을 활용한 첫 번째 워크플로우였습니다. n8n은 보통 클라우드에 호스팅해서 많이 사용하는 것으로 알고 있지만, 저는 우선 로컬 환경에서 워크플로우를 구성하며 전체 흐름을 확인했습니다.

가장 좋았던 점은 자동화 흐름을 눈으로 보면서 설계할 수 있다는 점이었습니다. 노드를 하나씩 연결하다 보면 데이터가 어디서 들어오고, 어디서 변환되고, 어디로 전달되는지 비교적 쉽게 파악할 수 있고, 흐름이 끊기거나 문제가 발생했을때 문제가 발생한 부분을 파악하기가 편하고, 워크플로우의 목표가 명확하다면 생산성이 좋다고 느꼈습니다.

다만 Notion API는 생각보다 Markdown과 바로 맞지 않는 부분이 많았습니다. Notion은 문서를 블록 단위로 잘게 나누어 관리하기 때문에, 각 블록 타입을 Markdown 문법으로 직접 매핑해야 했습니다. paragraph, heading, list, code, image, embed 같은 기본 블록은 처리할 수 있었지만, 복잡한 중첩 구조나 예외 케이스까지 안정적으로 다루려면 변환 로직을 더 다듬어야 합니다.

고도화하고 싶은 부분도 몇 가지 있습니다.

먼저 예외 처리를 보강해야 합니다. GitHub 업로드 실패, 이미지 다운로드 실패, Notion 블록 변환 실패 같은 상황이 생겼을 때 현재는 흐름을 추적하기가 쉽지 않습니다.

이미지 처리 로직도 더 정리하고 싶습니다. 지금은 이미지 URL 추출, 파일명 생성, 다운로드, GitHub 업로드, Markdown 치환 과정이 여러 노드에 나뉘어 있습니다. 동작은 하지만, 나중에 수정하려면 흐름을 다시 따라가야 합니다. 이미지 정보를 하나의 구조로 관리하고, 최종 Markdown 치환까지 더 단순하게 묶을 수 있을 것 같습니다.

이번 워크플로우는 Notion을 CMS처럼 쓰고 GitHub Pages를 배포 채널로 연결한 작업이었습니다. 완성도 높은 자동화라고 하기에는 아직 다듬을 부분이 있지만, 반복 작업을 줄이는 데는 충분히 효과가 있었습니다. 앞으로는 에러 처리와 이미지 처리 흐름을 정리해서, 실제 블로그 운영에 더 안정적으로 사용할 수 있도록 개선해볼 생각입니다. 또한, 깃허브 블로그뿐만이 아니라, naver-blog, medium, tistory와 같은 여러 블로그 포스팅 플랫폼에 fan-out하는 구조까지 가면 완성도 높은 자동화가 될 것 같습니다.

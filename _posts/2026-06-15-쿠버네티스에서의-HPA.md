## HPA

쿠버네티스에서 트래픽 버스트 상황이나 특정 Node, Pod에 부하가 쏠릴경우 컨트롤플레인은 자동으로 오토스케일링 합니다. 쿠버네티스에서는 크게 두가지 오토스케일러가 있는데요. 바로 HorizontalPodAutoSclaer(HPA)와 VerticalPodAutoscaler(VPA)가 있습니다. 이름 그대로 HPA는 부하(CPU 사용량, 메모리 사용량등)수평적으로 deployment를 통해 Pod 규모를 증설(Scale Out)하는 것이고, VPA는 수직적으로 하드웨어 스펙을 설정된 값으로 증설하는것(Scale up)입니다(Deployment, StatefulSet). 그 중에서도 오늘은 HPA에 대한 내용을 중심으로 알아보겠습니다.

부하가 다시 줄어들면, 오토스케일러는 이에 반응하여 다시 규모를 줄입니다.

### HPA Control Loop

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

그림 3: HPA의 ControlLoop (출처: NotebookLM)

HPA는 항상 MetricsServer와 함께 동작합니다. HPA의 기본 동작은 주기적으로(default interval: 15초) metricsserver에서 deployment들의 cpu utilization을 수집해온 다음, 설정된 목표 cpu 임계치를 넘었을때 scailing을 진행합니다. 이 설정값은 kube-controller-manager에서 horizontalp-pod-autoscaler-sync-period 파라미터를 수정할 수 있습니다.

> Once during each period, the controller manager queries the resource utilization against the metrics specified in each HorizontalPodAutoscaler definition. The controller manager finds the target resource defined by the scaleTargetRef, then selects the pods based on the target resource's .spec.selector labels, and obtains the metrics from either the resource metrics API (for per-pod resource metrics), or the custom metrics API (for all other metrics).

쿠버네티스 공식 문서

각 주기 동안 Controller-manager는 각 HPA 정의에 지정된 메트릭을 기준으로 리소스 활용도를 쿼리하고 ScaleTargetRef에 정의된 대상 리소스를 찾아 리소스의 .spec.selector 레이블을 기반으로 Pod를 선택하여 리소스 메트릭 API에서 메트릭을 가져옵니다.

1. kubelet이 cAdvisor를 통해 Pod의 CPU, Memory 등의 리소스를 수집. 

1. Metrisc Server가 Kubelet 데이터를 집계해서 /apis/metris.k8s.io AP로 노출

1. kube-controller-manager → 내부에서 HPA Controller Loop를 15초 주기로 실행

1. HorizontalPoadAutoScaler → 정책과 목표값 정의, 알고리즘으로 desireReplicas 계산

1. kube-apiserver → 상태 저장 및 Deployment에 PATCH 전달

1. Deployment → replicas 값 수신 후 ReplicaSet 조정

### Evaluation CPU resorce Algorithms 

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

그림4. 알고리즘의 동작 방식 (출처: NotebookLM)

알고리즘은 Pod들의 desired cpu resource를 분모로 두고, current cpu state를 분자로 둔 다음 백분율 하여 반올림 처리하여 최종 desiredReplica를 뽑아냅니다. 이 부분들을 자세히 한번 뜯어보겠습니다.

$$
desiredReplicas=ceil[currentReplica *\frac{curremtMetrivValue}{desidMetrivValue}]
$$

cpu utilization은 mm core 단위로 계산됩니다. 예를 들어 1core는 1000mm core입니다.

백분율은 이 millicore를 기준으로 계산됩니다.

현재 metric value가 200millicore이고 desired value가 100millicore일 때, 최종 desired replicas 200 / 100 = 2.0 이 됩니다. 즉, 현재 replica의 두배로 스케일링 합니다.

이번에는 metric value가 50millicore이면 50 / 100 = 0.5 replica, 올림처리되어 1이  되며  이 경우에는 스케일링을 진행하지 않습니다.

```
stabilizedDesiredReplicas =
    max(last_30_seconds_recommendations)
```

이면 max(10, 8, 5) = 10이 선택됩니다.

/assets/

HPA 는 desired replica를 계산하기 전에 아래와 같은 과정을 거칩니다.

먼저 Tolerance를 적용하기 전에 Pod 상태를 점검합니다.

- 메트릭이 없는 Pod가 있는가?

- Ready 상태가 아닌 Pod가 있는가?

- 종료(Terminating) 중인 Pod가 있는가?

- Failed Pod가 있는가? 

이러한 Pod 들을 어떻게 취급할지 결정한 뒤 최종 replica 수를 계산합니다.

Resource metric (CPU/Memory)인 경우

예를 들어 CPU 기반 HPA라고 가정합니다.

```
metrics:
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 50
```

현재 상태

```
Pod-A Ready      CPU=80%
Pod-B Ready      CPU=90%
Pod-C Terminating
Pod-D Failed
```

Pod C의 상태를 보면 아래와 같은 deletionTimestamp가 들어가있습니다.

```
metadata:
  deletionTimestamp: "2026-06-14T10:00:00Z"
```

이 상태는 kubectl delete pod Pod-c가 호출된 상태입니다.

하지만 아직 프로세스는 살아 있을 수 있습니다.

```
SIGTERM -> graceful shutdown -> Termination 
```

하지만 HPA는 이 Pod는 곧 사라질 Pod라고 판단하고 CPU 계산식에서 제외합니다. 삭제될 Pod의 낮은 CPU 때문에 Scale-out이 방해되면 안되므로 제외합니다.

같은 논리로 Failed Pod또한 무시합니다. Failed 상태는 OOMKilled, ContainerCrash, ImagePullBackOff, NodeLost 등으로 인해 상태가 Failed로 확정된 Pod를 의미합니다. 이미 서비스 대상 중인 Pod가 아니므로 CPU 계산식에서 제외합니다.

External/Object Metric인 경우는 Metric 값 + 현재 서비스 중인 Pod수를 계산해야 합니다.

여기서 External Metric은 외부 aws SQS Queue Length, kafka Log, HTTP Request Count와 같은 서비스를 의미합니다.

```
metrics:                  metrics:
- type: External          - type: Object
```

예를 들어 Queue Length = 1000, Pod = 5 라면 1000 / 5 = 200.

Pod당 200개의 작업을 하고 있다고 볼 수 있습니다.

중요한 부분은 Teraminating + Ready는 포함합니다.

```
Pod1 Ready
Pod2 Ready
Pod3 Ready + Terminating
```

삭제 명령이 내려졌으나, terminationGracePeriodSeconds=30 과 같은 설정값이 설정되어 있다면 30초동안 계속 요청을 받을 수 있습니다. 즉, 서비스 가능한 상태이기 때문에 HPA는 이 Pod가 완전히 종료될때까지, 그러니까 30초가 지날때까지 이 Pod를 함께 계산합니다.

이 부분을 조금만 더 살펴보자면, k delete pod <appname>과 같은 명령이 api-server를 통해  Pod에 전달되고, etcd는 Pod 객체에 deletionTimestamp가 즉시 반영합니다. 하지만 Pod Object는 아직 존재합니다. 단지 삭제 예정(marked for deletion) 상태가 된 것입니다.

이후에 Control Plane은 이 Pod는 곧 삭제될 Pod라고 판단합니다. 삭제될 Pod의 ip가 10.244.1.12라면  따라서 먼저 Service Endpoint에서 제거됩니다. 따라서 더 이상 새로운 트래픽을 받을수 없습니다.

EndPointList=[10.244.1.10, 10.244.1.11, 10.244.1.12]

deleteafterEndPointList=[10.244.1.10, 10.244.1.11]

node위의 Kubelet은 Pod 삭제를 감지하고 종료 요청을 컨테이너 런타임에 전송합니다. 여기서 중요한건 signal이 Kill이 아니라는 점입니다.

kubelet은 컨테이너런타임인 PID에게 SIGTERM signal을 전달합니다. 이후 pod위에 있는 nginx, spring boot, python과 같은 프로세스가 종료 준비를 합니다. DB Connection Close, 남은 Request 처리, 메모리 정리, 로그 Flush 등을 수행합니다.

이후에 Grace Period에 설정된 30초가 만료되면 SIGKILL signal을  전송하여 강제 종료하며 노드의 커널이 해당 Pod cgoroup 전체에 SIGKILL signal로 강제 종료하여 메모리를 회수합니다.

이때 containerd가 종료되었다고 판단하고 메모리 해제, 소켓 해제, 파일 핸들을 해제합니다. 이후 kubelet이 API server에 종료 사실을 알리고, control plane은 etcd에 Pod object를 삭제합니다. 이 시점에서는 k get pod 조회가 되지 않습니다.

```
kubectl delete
↓
deletionTimestamp 설정
↓
Terminating 상태
↓
Endpoint 제거
↓
preStop
↓
SIGTERM
↓
애플리케이션 정리
↓
프로세스 종료 (kill cgroup)
↓
Pod Object 삭제
```

### 노이즈 리소스 제외 

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

그림5. 사이드카 노이즈 제거 (출처: NotebookLM)

HPA 입장에서 모든 이벤트들이 시그널은 아닐겁니다. 예를 들어 workload와 무관한 부하들은 대부분 노이즈로 간주합니다.

여기서 워크로드와 무관한 부하들은 시스템 레벨에서 메모리 누수나 GC에서 발생하는 stop-the-world, 로그 폭발과 같은 것들을 의미합니다. 이런 문제들은 Pod를 늘린다고 해결되는것이 아닙니다. 노드레벨에서의 해결해야할 문제입니다.

SRE나 시스템 엔지니어, 클라우드 엔지니어 입장에서는 이 신호들이 중요하지만 HPA는 트래픽, 예를 들어 QPS나 RPS와 같은 지표에 맞게 스케일링하여 서비스에 장애가 나지 않게하는것이 목적이기 때문에 중요하지 않으므로 해당 메트릭들은 제외시키는게 좋습니다.

### Readiness와 함정: Warm-up Spike

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

그림 6: Warm-up Spike (출처: NotebookLM)

Pod Readiness에서 말하는 warm-up-spike는 Pod가 Ready 된 직후에 애플리케이션이 아직 완전히 예열되지 않아 CPU, 메모리, GC, JIT, Cache, Connection Pool, Thread Pool과 같은 자원이 갑자기 튀는 현상입니다. 즉, 트래픽을 받기 시작한 시점과 실제로 안정적으로 처리할 수 있는 시점 사이의 간극에서 생기는 스파이크 입니다. Pod는 Ready인데, 애플리케이션이 아직 Cold인 상태를 의미합니다.

예를 들어 JVM 애플리케이션이라면 이런 일이 흔합니다.

- 클래스 로딩과 JIT 컴파일이 몰린다

- 캐시가 비어 있어 DB/Redis 조회가 급증한다.

- 커넥션 풀이 아직 덜 채워져서 첫 요청들이 느려진다.

- 첫 요청에서 GC가 갑자기 급증한다.

이것이 왜 문제가 될까요? Ready 판정은 기본적으로 Service endpoint에 들어갈지 말지를 결정합니다. 그런데 Pod가 Ready가 된 직후 warm-up spike가 발생하면 첫 요청 지연이 커지고, HPA가 CPU spike를 보고 불필요하게 scale out할 수 있습니다.이러한 문제들을 Thrashing과 flapping이라고 합니다.

### Thrashing과 flapping

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

그림 7: flapping preventing startegy: 300seconds Stabilization window (출처: NotebookLM)

쿠버네티스에서 Thrashing이란 불필요하게, 그리고 연속적으로 스케일 인,아웃을 반복하여 리소스를 갉아먹는 현상을 의미합니다. 가상 메모리에서 말하는 Thrashing과는 조금 다릅니다만 리소스 낭비 현상을 의미하는것은 똑같습니다.

Thrashing을 설명하기 위해 다시 Spring boot기반 JVM 사례를 가져와보겠습니다. 이제 막 Pod를 시작하고 conainer에 spring boot가 부트로딩되고 있는 상황이라고 가정해봅시다.

timestamp: 2026-06-15-22:30:05

- Pod A : 90%

- Pod B: 85% 

- 평균 cpu 부하: 87.5%

HPA 계산(desired replica) = 2 * (87.5 / 70) = 2.5 = 3

2 → 3 개로 Scale Out.

timestamp: 2026-06-15-22:10

- Pod A: 50%

- Pod B: 55%

- Pod C: 20%

- 평균 CPU 부하: 41%

HPA 계산(desired replica) = 3 * (41 / 70) = 1.75 = 2

3 → 2 Scale Down

timestamp: 2026-06-15-22:22

- Pod A: 85% 

- Pod B: 88%

- 평균 cpu 부하: 86.5%

다시  2 → 3 scale out. 2 → 3 → 2 → 3 → 2 → 3 무한 반복. 이것이 쓰레싱입니다. 특히 JVM 같은 경우 새 Pod가 생기면 즉시 성능을 내지 못합니다. 예를들어 spring boot.

create new pod.

1. Container Start

1. JVM Start

1. Class Loading

1. Spring Context initialize

1. JIT Warmup

1. Ready

JVM에서는 또한 GC가 동작하면 같은 문제로 CPU 부하가 치솟을 수 있으며 마찬가지로 HPA가 이에 반응하여 scale out을 하고 GC가 끝나면 scale down하는 일이 반복될 수 있습니다.

가장 대표적인 대처 방법은 damping 입니다. 변화에 둔감하게 한다라는 용어인데, 쿠버네티스에서스는 애플리케이션 warm-up시 짧은 진동구간을  완화시킨다는 의미로 쓰입니다. Pod Ready를 delay, stabilizatin window, readiness probe 설정과 같은 방법으로 이를 달성합니다.

이전에 replica 수를 Downscale했던 방법처럼 scale-up policy를 Stabilization Window의 시간값을 조정합니다. 기본값은 300(5분) 입니다. startup시에는 max-replica와 같은 up-scale 정책과 maxSurge,maxUnavailable, minReadySeconds와 같은 롤링 업데이트 정책을 보수적으로 설정하는것도 좋은 방법입니다.

또 다른 방법은 애플리케이션이 진짜로 준비되기 전에는 readiniess probe를 통과시키지 않게 합니다. cache preload가 완료되거나 DB Connection Success, 내부 의존성 확인 완료 후에 Ready로 바꿀수 있게 합니다. 컨테이너가 뜬 순간과 트래픽을 받아도 되는 순간을 분리하는것이 핵심입니다.

startupProbe를 사용하는것도 좋은 방법일것 같습니다. JVM, Spring Boot와 같은 대형 모델 로딩처림 시작 시간이 긴 서비스에는 특히 더 유용합니다.

- startupProbe: 앱이 완전히 뜰 떄까지 대기

- readinessProbe: 트래픽을 받아도 되는지 판단

- livenessProbe: 죽었는지 판단

이렇게 분리하면 시작 중에는 liveness가 괜히 죽이지 않고 실제 준비 여부를 더 정확히 반영할 수 있습니다.

### Flapping

/assets/image_214d9ce0-d601-4b0f-9841-118cff09f233.png

flapping은 시스템 상태가 두 개 이상의 상태 사이를 짧은 시간동안 계속 왔다갔다 하는 현상을 의미합니다. 원래는 네트워크 장비나 모니터링 분야에서 많이 사용하던 용어인데 쿠버네티스에서도 많이 사용됩니다.

CPU 사용률이 다음과 같다고 해봅시다.

임계값 = 70%

Pod1: 68%
Pod2: 72%
Pod3: 69%
Pod4: 71%
Pod5: 68%
Pod 6: 73%
Pod 7: 69%

cpu 리소스가 위처럼 경계선 근처를 계속 왔다갔다 하면 HPA는 2 pod → 3 pod → 2 pod 처럼 반복적으로 scale-out, scale down을 반복할것입니다. 이것이 scaoling flapping 현상입니다.

잘 생각을 해보면, sacle-out, down 자체에도 리소스가 소모됩니다. 이것이 지속적으로 반복된다면 cpu 리소스가 심하게 낭비될것입니다. 또는 Pod Readniess 자체에도 발생합니다.

Rediness Probe 상태에도 이는 적용됩니다.

Sucess, Failure, Success, Failure가 반복되면, Endpoint 추가 → 제거 → 추가 → 제거가 반복되고 이는 서비스 관점에서는 트래픽 전달 → 전달 중단 → 전달 → 중단으로 이어질것입니다. 이를 Readiness flapping이라고 부릅니다.

thrashing 방지와 마찬가지로 완전히 warm up이 끝나고 난 후의 metric을 신뢰하도록 하고 스케일 아웃 다운전략을 이 stabilization window동안 보수적으로 취하도록 하는것이 좋습니다. 이에 기본적으로 HPA Tolerance라는 옵션을 제공합니다.

$$
ratio = \frac{currentMetricValue}{desiredMetricValue}
$$

|1 - ratio| < tolerance 이면 스케일링을 하지 않습니다. 기본값은 0.1 즉 10% 입니다.

Target CPU = 70%, Tolerance = 10%

허용 범위 = 70 * 0.9 = 63% , 70 * 1.1 = 77%

즉, 10% 오차를 허용하여 63 ~ 77% 구간에서는 Pod수를 변경하지 않습니다.

그래서 Tolerance는 제어공학 관점에서 보면 컨트롤러의 민감도를 낮추는 데드밴드(Deadband) 역할을 하며, HPA가 CPU 사이클의 작은 흔들림에 반응하지 않도록 만드는 첫 번째 방어선이라고 볼 수 있습니다.

혹은 CPU 지표 자체를 불신하고 QPS, Queue Length, response Latency 지표만 metric으로 설정해두는것도 좋은 방법일 것 같습니다.

실제 대규모 Hyper-Pod 환경에서는 사용자 정의 기반 비즈니스 지표 (SQS Queue Length, Request Concurrency, GPU Queue, Lag)등으로 스케일링 하는 경우가 많습니다. 단순히 CPU 사용량만으로는 부하 측정이 적합하지 않을때가 많기 때문이기도 한데요. 예를 들어 Node.js로 웹 서비스를 운영하는 서버가 있다고 해봅시다.

- CPU Utilization: 30%

- Request Queue: 500

상황일때, CPU 사용량은 한가하지만 실제로 request queue에는 500개가 쌓여있습니다. 단순히 CPU 사용량만 보고 스케일링을 진행한다면 현재 빠르게 증설해야 하는 상황에서 반응하지 못할겁니다. 따라서 웹 서비스에서는 특정 시간 윈도우동안 요청 횟수, 티켓 갯수, 혹은 패킷 갯수나 대기열의 길이나 네트워크 지연시간, QPS, RPS를 기반으로 설정하는 경우가 많습니다.

실무에서는 HPA를 CPU만 보는 장치로 쓰기보다, 워크로드별 "진짜 병목"을 보는 오토스케일러로 씁니다. Kubernetes HPA는 기본적으로 metrics.k8s.io(리소스), custom.metrics.k8s.io, external.metrics.k8s.io를 통해 CPU/메모리뿐 아니라 커스텀·외부 메트릭까지 받을 수 있습니다.

가장 흔한 패턴은 웹/API 서비스입니다. 이때는 cpu, memory 같은 리소스 메트릭에 더해 request concurrency, P95 latency 같은 커스텀 메트릭을 붙여서 "사용자 요청량"에 반응하게 만듭니다. HPA는 Pod별 리소스 사용량을 평균내거나, 커스텀/외부 메트릭을 조회해서 원하는 replica 수를 계산합니다.

비동기 워커는 큐 길이가 가장 좋은 신호입니다. 예를 들어 SQS 기반 워커는 KEDA의 AWS SQS scaler를 많이 쓰는데, 기본적으로 ApproximateNumberOfMessages + ApproximateNumberOfMessagesNotVisible를 "actual messages"로 삼아 스케일합니다. 즉 "밀린 일감"을 직접 보고 늘리는 방식입니다.

GPU 추론은 보통 GPU utilization을 직접 HPA에 넣기보다, DCGM Exporter로 GPU 메트릭을 Prometheus에 내보내고 그걸 HPA/KEDA의 custom metric으로 씁니다. NVIDIA 문서는 DCGM Exporter가 GPU telemetry를 /metrics로 노출하고 Prometheus가 스크랩할 수 있다고 설명합니다. KServe의 LLM 예시에서는 vllm:num_requests_running을 보고 동시 요청 수가 임계값(예: 2)을 넘으면 KEDA가 Pod 수를 늘립니다.

클라우드 네이티브 ML/LLM 쪽에서는 "GPU 사용률"보다 "큐 깊이, 동시 요청 수, cache residency, SLA"가 더 좋은 신호인 경우가 많습니다. KServe와 llm-d는 LLM 워크로드를 Kubernetes-native하게 다루면서 GPU utilization, queue depth, cache residency, SLA constraints, load distribution 같은 신호를 함께 고려한다고 설명합니다.

MemoryPressure 이벤트는 HPA 신호가 아니라 노드 레벨 신호입니다. Kubernetes 문서는 MemoryPressure가 노드 메모리 부족을 뜻하며, kubelet이 이런 상황에서 Pod를 강제 종료(eviction)할 수 있다고 설명합니다. 즉 이건 "replica를 늘려서 해결"할 문제가 아니라, 노드/클러스터 레벨에서 대응해야 하는 시그널입니다.

swap 사용량도 공식 HPA의 대표 built-in 신호는 아닙니다. 실무에서는 이런 값도 Prometheus나 KEDA의 custom/external metric으로 노출해서 쓸 수는 있지만, 보통은 먼저 "이게 진짜 workload 신호인가, 아니면 노드/애플리케이션 이상 징후인가"를 구분합니다. HPA가 커스텀·외부 메트릭을 지원하고, KEDA가 Prometheus 쿼리를 스케일 신호로 쓸 수 있다는 점이 그 기반입니다.

HyperPod 쪽은 두 층으로 보는 게 좋습니다. Pod 레벨에서는 HyperPod inference operator의 autoScalingSpec이나 KEDA ScaledObject로 CloudWatch, Amazon Managed Prometheus, SQS, Prometheus, CPU/Memory를 기반으로 스케일합니다. 노드 레벨에서는 SageMaker HyperPod가 EKS용 managed Karpenter 기반 node autoscaling을 제공합니다. AWS 문서에는 minReplicaCount: 0, cooldownPeriod, scaleDownStabilizationTime 같은 설정으로 0 Pod까지 내릴 수 있는 예시도 나옵니다.

실무 관점의 정리는 이렇습니다. 웹/API는 CPU+latency, 비동기 작업은 queue length, LLM/GPU 추론은 concurrent requests + queue depth + GPU telemetry, 노드 건강은 MemoryPressure 같은 node condition, 대규모 AI 클러스터는 pod autoscaling과 node autoscaling을 분리해서 봅니다.
